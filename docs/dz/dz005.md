# Нагрузочное тестирование

## Цель

- Перейти от вопроса "работает ли это?" к вопросу "насколько хорошо это работает?".
- Освоить профессиональный инструмент для нагрузочного тестирования (NBomber) в привычной C# среде.
- Получить первые количественные метрики производительности (RPS, latency) для своего сетевого сервиса.
- Научиться писать нагрузочные тесты для кастомных протоколов, а не только для стандартных HTTP API.

## Результат

Студент создаст отдельный проект для нагрузочных тестов в своем решении. 
Внутри будет написан и сконфигурирован сценарий на NBomber, который имитирует работу множества клиентов с сервером из ДЗ №4. 
По итогу студент получит отчет с реальными показателями производительности своего приложения.

## Зачем это нужно

Это задание учит разработчика мыслить как инженер по производительности. 
Без нагрузочного тестирования любые заявления о "высокой производительности" являются лишь догадками. 
Это ДЗ дает студенту инструмент для получения объективных доказательств и учит его находить реальные пределы своего приложения, что является обязательным навыком при разработке любых серьезных backend-систем.


## Описание/Пошаговая инструкция выполнения домашнего задания

### Пункт 1. Настройка проекта для нагрузочного тестирования

Создайте в своем решении новое консольное приложение. 
Добавьте в него NuGet-пакет NBomber. 
Этот проект будет содержать ваши нагрузочные тесты.

### Пункт 2. Создание простого TCP-клиента

Внутри тестового проекта создайте простой класс-клиент, который будет взаимодействовать с вашим сервером. 
У него должны быть методы, например, ConnectAsync(), SetAsync(string key, byte[] value) и GetAsync(string key). 
Эти методы должны инкапсулировать логику отправки команд по TCP и чтения ответов.

### Пункт 3. Реализация шага теста (Step)

Используя NBomber, создайте шаг теста (Step.Create). 
Внутри этого шага должен создаваться экземпляр вашего TCP-клиента. 
Шаг должен выполнять одну простую операцию, например, вызывать client.SetAsync() со случайным ключом и значением. 
Шаг должен возвращать Response.Ok() в случае успеха или Response.Fail() в случае ошибки.

### Пункт 4. Создание сценария нагрузки (Scenario)

Создайте сценарий (ScenarioBuilder.CreateScenario), который будет использовать шаг, созданный в предыдущем пункте. 
Сконфигурируйте сценарий так, чтобы он имел фазу "разогрева" (warm-up) в 5-10 секунд и основную фазу нагрузки. 
Используйте Simulation.Inject для имитации нагрузки, например, 100 запросов в секунду в течение 30 секунд.

### Пункт 5. Запуск и анализ результатов

Запустите ваш сервер из ДЗ №4, а затем запустите проект с нагрузочным тестом. 
Проанализируйте отчет, который NBomber выведет в консоль. 
Обратите внимание на ключевые показатели: количество выполненных запросов (Request Count), RPS (запросов в секунду), количество ошибок (Fail Count) и время отклика (latency), особенно p95 и p99.


## Критерии оценки

- Создан и настроен отдельный проект для тестов с установленным пакетом NBomber - 1 балл
- Реализован простой TCP-клиент, инкапсулирующий логику отправки команд и чтения ответов - 2 балла
- Корректно реализован шаг теста в NBomber, который использует TCP-клиент для выполнения одной операции - 3 балла
- Создан и сконфигурирован сценарий нагрузки с фазами разогрева и основной нагрузки - 2 балла
- Тест успешно запускается, и студент может проинтерпретировать базовые результаты из отчета NBomber (RPS, ошибки) - 2 балла
