# Работа со сложными объектами и стандартная сериализация

## Цель

- Перейти от хранения примитивных типов (byte[]) к работе со структурированными, сложными объектами (DTO).
- Освоить стандартную для индустрии библиотеку System.Text.Json для сериализации и десериализации объектов.
- Понять, как сериализация влияет на протокол взаимодействия и производительность клиент-серверного приложения.
- Подготовить приложение к следующему этапу — замене стандартного сериализатора на кастомный, высокопроизводительный.

## Результат

Студент получит обновленную версию своего сервера, который теперь оперирует не просто массивами байт, а полноценными C# объектами. 
Сервер сможет принимать объект в виде JSON-строки, десериализовать его, сохранить в бинарном виде и при запросе отдать обратно.

## Зачем это нужно

Реальные приложения редко работают с "сырыми" байтами. Они оперируют моделями данных (пользователи, заказы, товары). 
Это задание переводит "игрушечный" проект на следующий уровень, приближая его к реальным задачам. 
Оно также создает "точку отсчета" производительности, которую мы будем кардинально улучшать в следующем ДЗ с помощью генераторов кода.


## Описание/Пошаговая инструкция выполнения домашнего задания

### Пункт 1. Создание модели данных (DTO)

Создайте простой public class UserProfile. 
У него должно быть как минимум три свойства разных типов, например: 
- public int Id { get; set; }, 
- public string Username { get; set; }, 
- public DateTime CreatedAt { get; set; }. 
Этот класс будет нашей моделью данных.

### Пункт 2. Модификация интерфейса SimpleStore
Измените сигнатуры методов в вашем классе SimpleStore. 
Теперь они должны работать с UserProfile, а не напрямую с byte[]:
- void Set(string key, UserProfile profile)
- UserProfile Get(string key)
Внутреннее хранилище Dictionary<string, byte[]> остается без изменений.

### Пункт 3. Реализация сериализации и десериализации

Внутри метода Set используйте System.Text.Json.JsonSerializer.SerializeToUtf8Bytes(profile) для преобразования объекта UserProfile в byte[] перед сохранением в словарь. 
Внутри метода Get после получения byte[] из словаря используйте JsonSerializer.Deserialize(bytes) для преобразования данных обратно в объект.

### Пункт 4. Обновление логики TCP-сервера

Модифицируйте метод ProcessClientAsync в TcpServer. 
Теперь, когда приходит команда SET, ее VALUE часть является JSON-строкой. 
Вы должны взять этот ReadOnlySpan, представляющий JSON, и десериализовать его в объект UserProfile (JsonSerializer.Deserialize(valueSpan)), прежде чем вызывать метод _store.Set().

### Пункт 5. Обновление клиента для нагрузочного теста

Модифицируйте TCP-клиент, который вы использовали в ДЗ №5. 
Теперь при отправке команды SET он должен сначала создать экземпляр UserProfile, сериализовать его в JSON-строку, а затем отправить эту строку как VALUE часть команды. 
Это необходимо, чтобы проверить работоспособность всей цепочки.


### Критерии оценки

- Создан класс-DTO UserProfile с несколькими свойствами разных типов - 1 балл
- Интерфейс SimpleStore изменен для работы с UserProfile, а не с byte[] - 2 балла
- В SimpleStore корректно реализована логика сериализации в Set и десериализации в Get с помощью System.Text.Json - 3 балла
- Логика TcpServer обновлена для десериализации входящего JSON-значения при выполнении команды SET - 2 балла
- Клиентская часть (например, для нагрузочного теста) обновлена для отправки сериализованных в JSON объектов, что доказывает работоспособность всей системы - 2 балла
