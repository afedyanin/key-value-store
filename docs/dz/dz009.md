# Продвинутая отладка: анализ сбоев

## Цель

- Применить на практике ультимативные инструменты для диагностики сложных проблем: WinDbg и расширение SOS.
- Научиться проводить post-mortem анализ ("вскрытие") приложений по дампам памяти, снятым в production-среде.
- Диагностировать трудноуловимые многопоточные проблемы, такие как взаимоблокировки (deadlocks), которые не видны обычным профилировщикам.
- Развить навык не написания кода, а его глубокого анализа в самых критических ситуациях.

## Результат

Студент предоставит не код, а письменный отчет о расследовании. 
В этом отчете будет детально описана причина "зависания" предоставленного приложения, основанная на анализе дампа памяти. 
Отчет будет содержать информацию о вовлеченных потоках и примитивах синхронизации, вызвавших проблему.

## Зачем это нужно
Это задание имитирует работу инженера по надежности (SRE) или эксперта из команды поддержки, которому принесли самую сложную проблему из продакшена. 
Умение анализировать дампы и находить корень проблемы, когда приложение уже "упало" или "зависло", — это один из самых ценных и редких навыков, который отличает эксперта, способного поддерживать критически важные системы.


## Описание/Пошаговая инструкция выполнения домашнего задания:

### Пункт 1. Настройка окружения для отладки

Установите необходимые инструменты для продвинутой отладки. 
Установите WinDbg Preview из Microsoft Store. 
Установите глобальный инструмент dotnet-dump с помощью команды dotnet tool install -g dotnet-dump.

### Пункт 2. Воспроизведение проблемы и создание дампа памяти

В занятии по WinDBG есть проект консольного приложения, в котором намеренно создана ситуация взаимоблокировки (deadlock), возьмите его или создайте своё. 
Запустите это приложение. 
Убедитесь, что оно "зависло" и не завершает работу. 
Определите его Process ID (PID) с помощью Диспетчера задач или команды dotnet-dump ps. 
Создайте дамп памяти этого процесса с помощью команды dotnet-dump collect -p [PID].

### Пункт 3. Анализ потоков и стеков вызовов

Запустите WinDbg и откройте в нем созданный файл дампа. 
Загрузите расширение для отладки .NET — SOS (команда .loadby sos coreclr). 
Используйте команду !threads, чтобы увидеть список всех управляемых потоков в приложении. 
Изучите стеки вызовов нескольких потоков с помощью команды !clrstack, чтобы понять, чем они заняты.

### Пункт 4. Диагностика взаимоблокировки (Deadlock)

Используйте специализированную команду SOS для поиска дедлоков: !syncblk. 
Проанализируйте вывод этой команды. 
Она должна показать, какие потоки ожидают какие блокировки, образуя замкнутый круг.

### Пункт 5. Написание отчета о расследовании

На основе анализа, проведенного в WinDbg, напишите краткий отчет в текстовом файле. 
Отчет должен отвечать на следующие вопросы:
- В чем заключалась проблема (например, взаимоблокировка)?
- Какие потоки (указать их ID из вывода !threads) были вовлечены в проблему?
- Какие именно примитивы синхронизации (например, lock на каких объектах) привели к взаимоблокировке?
Приложите к отчету вывод команд.


## Критерии оценки

- Установлены и готовы к работе WinDbg и dotnet-dump - 1 балл
- Успешно создан дамп памяти "зависшего" приложения с помощью dotnet-dump - 2 балла
- В WinDbg выполнен базовый анализ: загружен SOS, получен список потоков и проанализированы их стеки вызовов - 2 балла
- С помощью команды !syncblk корректно определена причина зависания (взаимоблокировка) - 3 балла
- Составлен четкий письменный отчет, который правильно идентифицирует вовлеченные в дедлок потоки и объекты блокировки - 2 балла
